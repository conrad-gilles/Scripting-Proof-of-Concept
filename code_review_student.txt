========== ContextFactories.cs

A factory should be registered via DI.
So the create method does not need passed a IServiceProvider serviceProvider. The factory class can simply get the relevant services from via the Constructor.

Example:

    public class ContextFactory(ConsoleLogger consoleLogger, DataAccess dataAccess) : Ember.Scripting.IContextFactory
    {
        public static GeneratorContextSF Create(LabOrder labOrder, Patient patient)
        {
            return new GeneratorContext(labOrder, patient, logger, data);
        }
    }
	
You also mixed DI-services with actual parameters.
For example LabOrder and Patient are POCOs not services. They are parameters that need to be provided by the developer when they use the factory. On the other hand ConsoleLogger and DataAccess are services that need to be resolved from DI via the constructor.

There is no need for a IContextFactory interface.
This is because each factory, like GeneratorContextFactory defines its own parameters that are different from the factory of another context. So the interface cannot generically express parameters they expect.

You can have an interface per factory like IGeneratorContextFactory. As this allows for unit test mocking etc...

So:

    public interface IGeneratorContextFactory()
    {
        public static GeneratorContextSF Create(LabOrder labOrder, Patient patient);
    }
